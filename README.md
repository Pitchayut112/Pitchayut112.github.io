# ระบบจองคิวไลฟ์สด (Live Booking System)

ระบบจองคิวไลฟ์สดที่กันการจองทับวันกัน, มีรหัสผ่านแยกตามลูกค้า, และซ่อนชื่อลูกค้าจากคนทั่วไปด้วยเอฟเฟกต์เบลอ

## ภาพรวมระบบ

| ไฟล์ | หน้าที่ |
|---|---|
| `index.html` | หน้าปฏิทินสาธารณะ — ใครก็เข้าดูได้ ล็อกอินเพื่อจองคิว |
| `admin.html` | หน้าแอดมิน — จัดการลูกค้า, เพิ่ม/แก้ไข/ลบการจอง |
| `assets/config.js` | ที่เก็บค่า Supabase URL/Key และรหัสผ่านแอดมิน |
| `assets/style.css` | สไตล์ทั้งหมด |
| `assets/app.js` | ลอจิกหน้าจองคิว |
| `assets/admin.js` | ลอจิกหน้าแอดมิน |
| `schema.sql` | คำสั่งสร้างฐานข้อมูลทั้งหมด (รันครั้งเดียวบน Supabase) |

**วิธีทำงานของการกันจองทับกัน:** ในตาราง `bookings` คอลัมน์ `booking_date` ถูกตั้งเป็น `unique` ที่ระดับฐานข้อมูล หมายความว่าต่อให้ลูกค้า 2 คนกดจองวันเดียวกันพร้อมกันเป๊ะ ฐานข้อมูลจะให้คนแรกผ่าน และคนที่สองจะได้รับข้อความ "วันนี้ถูกจองไปแล้ว" ทันที — ไม่มีทางจองทับกันได้แม้แต่กรณี race condition

---

## ขั้นตอนที่ 1: ตั้งฐานข้อมูล Supabase (ฟรี)

1. ไปที่ [supabase.com](https://supabase.com) แล้วสมัครสมาชิก (ใช้ GitHub login ได้)
2. กด **New Project** ตั้งชื่อโปรเจกต์อะไรก็ได้ เลือก region ใกล้ไทย (Singapore) แล้วตั้งรหัสผ่าน database (เก็บไว้เผื่อใช้)
3. รอประมาณ 1-2 นาทีให้โปรเจกต์สร้างเสร็จ
4. ไปที่แถบเมนูซ้าย เลือก **SQL Editor**
5. เปิดไฟล์ `schema.sql` ที่ให้มา **คัดลอกทั้งหมด** แล้ววางในช่อง SQL Editor
6. กด **Run** (หรือ Ctrl+Enter) — ถ้าสำเร็จจะเห็น "Success. No rows returned"

## ขั้นตอนที่ 2: เอา URL และ Key มาใส่ในเว็บ

1. ใน Supabase ไปที่ **Project Settings** (ไอคอนเฟือง) > **API**
2. คัดลอกค่า **Project URL** กับ **anon public key**
3. เปิดไฟล์ `assets/config.js` แล้วแก้:

```js
const SUPABASE_URL = "https://xxxxxxxxxxxx.supabase.co";   // ใส่ Project URL ของคุณ
const SUPABASE_ANON_KEY = "eyJxxxxxxxxxxxxxxxxxxxxxxxx";    // ใส่ anon public key

const ADMIN_PASSWORD = "ตั้งรหัสผ่านแอดมินของคุณเอง";
```

> ⚠️ **anon public key ปลอดภัยที่จะเปิดเผยบนเว็บ** เพราะถูกออกแบบมาให้ใช้ฝั่ง client อยู่แล้ว ส่วนรหัสผ่านลูกค้าจริงๆถูกซ่อนไว้ในฐานข้อมูล (RLS ปิดกั้นไม่ให้อ่านตรงๆ) เข้าถึงได้แค่ผ่านฟังก์ชัน login ที่เรานิยามไว้เท่านั้น

## ขั้นตอนที่ 3: เพิ่มลูกค้าคนแรก

1. เปิด `admin.html` ในเบราว์เซอร์ (เปิดจากไฟล์ตรงๆก็ทดสอบได้ก่อนอัปขึ้น GitHub)
2. กรอกรหัสผ่านแอดมินที่ตั้งไว้ในขั้นตอนที่ 2
3. กด **+ เพิ่มลูกค้า** กรอก username / รหัสผ่าน / ชื่อที่แสดง / เลือกสี แล้วบันทึก
4. ทำซ้ำสำหรับลูกค้าทุกคน — ส่ง username + password ให้ลูกค้าแต่ละคนเอง (เช่นทาง LINE)

## ขั้นตอนที่ 4: อัปขึ้น GitHub Pages

1. สร้าง repository ใหม่บน GitHub (Public ก็ได้ เพราะรหัสผ่านลูกค้าไม่ได้อยู่ในไฟล์ใดๆเลย)
2. อัปโหลดไฟล์ทั้งหมดในโฟลเดอร์นี้ขึ้น repo (ลาก-วางผ่านหน้าเว็บ GitHub ได้เลย หรือใช้ git push)
3. ไปที่ **Settings > Pages** ของ repo
4. หัวข้อ **Branch** เลือก `main` (หรือ branch ที่ใช้) แล้วกด **Save**
5. รอ 1-2 นาที จะได้ลิงก์แบบ `https://ชื่อของคุณ.github.io/ชื่อ-repo/`
6. แชร์ลิงก์นี้ให้ลูกค้าจองคิวได้ และเก็บลิงก์ `/admin.html` ไว้ใช้เองคนเดียว

> 💡 หน้า `admin.html` ไม่ได้ผูกกับสิทธิ์อะไรเป็นพิเศษ ใครรู้ลิงก์ตรงนี้ก็เปิดได้ แต่ต้องรู้รหัสผ่านแอดมินก่อนถึงจะเข้าจัดการได้ ถ้าอยากปลอดภัยขึ้นไปอีกขั้น แนะนำตั้งชื่อไฟล์ admin ให้เดาไม่ได้ เช่น `admin-x7k2.html` แทน

## วิธีใช้งานสำหรับลูกค้า

1. เข้าลิงก์เว็บ กด **เข้าสู่ระบบลูกค้า**
2. กรอก username / รหัสผ่านที่ได้รับ
3. เลือกวันที่ขึ้น "ว่าง" บนปฏิทิน จะมีหน้าต่างยืนยัน ใส่หมายเหตุได้ (ไม่บังคับ) แล้วกดยืนยัน
4. คิวของตัวเองจะโชว์เป็นชื่อจริงพร้อมสีประจำตัว ส่วนคิวของคนอื่นจะเห็นเป็นชื่อเบลอ
5. ถ้าต้องการยกเลิก/แก้ไข ต้องแจ้งแอดมินให้แก้ไขให้ (ลูกค้าทำเองไม่ได้ตามที่ออกแบบไว้)

## วิธีใช้งานสำหรับแอดมิน

1. เข้า `admin.html` กรอกรหัสผ่านแอดมิน
2. หัวข้อ **ลูกค้า** — เพิ่ม/แก้ไข/ลบลูกค้าได้ที่นี่ (ลบลูกค้าจะลบการจองของลูกค้านั้นไปด้วย)
3. หัวข้อ **การจองคิว** — ดูคิวทั้งเดือน เพิ่มคิวให้ลูกค้าได้โดยตรง, แก้ไขวันที่/ลูกค้า/หมายเหตุ, หรือลบคิวได้

## คำถามที่พบบ่อย

**Q: ถ้าลืมรหัสผ่านแอดมินทำยังไง?**
A: แก้ค่า `ADMIN_PASSWORD` ใน `assets/config.js` แล้วอัปขึ้น GitHub ใหม่ได้เลย

**Q: เปลี่ยนจาก "จองเป็นวัน" เป็น "จองเป็นช่วงเวลา" ได้ไหม?**
A: ได้ แต่ต้องแก้โครงสร้างตาราง `bookings` ให้มีคอลัมน์เวลาเริ่ม/จบ และเปลี่ยน constraint จาก unique date เป็นเช็คช่วงเวลาทับกัน (ซับซ้อนกว่านี้พอสมควร) — บอกได้ถ้าต้องการให้ปรับ

**Q: Supabase ฟรีจำกัดแค่ไหน?**
A: แผนฟรีรองรับโปรเจกต์ขนาดเล็ก-กลางได้สบายๆ (500MB database, ผู้ใช้งานหลักพันต่อเดือน) เพียงพอสำหรับลูกค้า 10-50 คนแน่นอน ระบบจะ pause อัตโนมัติถ้าไม่มีการใช้งานเกิน 1 สัปดาห์ แต่จะกลับมาทำงานเองทันทีที่มีคนเข้าใช้

**Q: ข้อมูลรหัสผ่านลูกค้าปลอดภัยแค่ไหน?**
A: รหัสผ่านเก็บเป็นข้อความปกติในฐานข้อมูล (ไม่ได้เข้ารหัสแบบ hash) ซึ่งเพียงพอสำหรับงานขนาดนี้เพราะ RLS ปิดกั้นไม่ให้ใครอ่านตารางนี้ได้โดยตรงอยู่แล้ว ถ้าต้องการความปลอดภัยสูงขึ้นไปอีก (เช่น เข้ารหัส bcrypt) แจ้งได้ จะปรับ schema ให้
